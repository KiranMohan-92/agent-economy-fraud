  <files>analysis/moltbook-platform-analysis.md</files>
  <action>
    Access Moltbook platform documentation.

    1. Locate Moltbook documentation (official site, API reference, or GitHub)
    2. Survey documentation structure focusing on:
       - Agent social platform features
       - Listing creation and management APIs
       - Transaction request/response flows
       - Social graph and connection mechanisms
    3. Extract specific APIs relevant to A2A social commerce:
       - Agent profiles and identity
       - Listing creation/management
       - Transaction initiation and acceptance
       - Social graph operations
    4. Document social and behavioral constraints:
       - Reputation scoring (if present)
       - Connection limits or requirements
       - Verification mechanisms
       - Rate limits on social interactions

    Use attack-chain-first approach: Focus on commerce-enabling features.
    Pay special attention to reputation systems as these are critical for Sybil analysis.

    If Moltbook documentation is inaccessible or incomplete:
    - Document the gap explicitly
    - Note which features must be inferred
    - Flag confidence reduction for affected analysis
  </action>
  <verify>
    1. Documentation access: Successfully located Moltbook docs or explicit gap documentation
    2. Coverage: Identified social features, listings, reputation mechanisms
    3. Traceability: Each finding includes specific documentation reference
    4. Gap honesty: Limitations explicitly documented
  </verify>
  <done>Moltbook documentation surveyed with social features, listing mechanisms, and reputation systems identified. All findings trace to documentation with explicit gap documentation.</done>

<task type="auto">
  <name>Task 2: Analyze reputation systems and Sybil attack vectors</name>
  <files>analysis/reputation-system-analysis.md</files>
  <action>
    Analyze Moltbook's reputation system for agent-specific vulnerabilities.

    1. Document reputation mechanisms:
       - How reputation scores are calculated
       - What actions affect reputation
       - How reputation is displayed/used
       - Any Sybil resistance features

    2. Map Sybil attack vectors:
       - Can agents create multiple identities?
       - How quickly can agents build reputation vs. humans?
       - Can reputation be transferred between agents?
       - What prevents or enables reputation gaming?

    3. Quantify scale advantage:
       - Human reputation building: typically weeks/months of consistent positive behavior
       - Agent reputation building: limited only by API rate limits and coordination
       - Calculate scale multiplier: (agent reputation speed / human reputation speed)

    4. Classify by detection difficulty:
       - Easy: Obvious Sybil patterns (multiple identities from same source)
       - Medium: Sophisticated identity separation with some behavioral differences
       - Hard: Distributed Sybil with realistic behavioral variation
       - Impossible: Perfect mimicry of human social patterns with genuine social graph integration

    5. Document reputation gaming implications:
       - How compromised reputation enables fraud
       - Detection latency for reputation attacks
       - Cross-platform reputation spillover (if applicable)

    Use detection difficulty as PRIMARY axis.
    Every Sybil vector must map to specific Moltbook platform features.
  </action>
  <verify>
    1. Platform grounding: All Sybil vectors map to specific Moltbook features
    2. Scale analysis: Numerical comparison of agent vs. human reputation building speed
    3. Detection difficulty: All vectors classified by detection difficulty
    4. Sybil resistance: Any platform protections documented (or lack thereof)
    5. Fraud linkage: Clear path from reputation compromise to fraud enablement
  </verify>
  <done>Reputation system analysis complete with Sybil attack vectors mapped to Moltbook features. Scale advantage quantified. All vectors classified by detection difficulty.</done>
</task>
