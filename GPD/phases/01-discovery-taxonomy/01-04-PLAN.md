  <name>Task 1: Search for public A2A transaction datasets</name>
  <files>analysis/data-acquisition-plan.md</files>
  <action>
    Conduct systematic search for public A2A transaction datasets.

    1. Search methodology (document each):
       - Kaggle: Search for "agent transaction", "multi-agent commerce", "automated trading", "bot transaction"
       - UCI ML Repository: Search for financial, transaction, fraud detection datasets
       - Google Dataset Search: "agent-to-agent", "autonomous agent transaction"
       - Academic sources: Check papers from literature survey for linked datasets
       - Platform APIs: Check if OpenClaw/Moltbook provide public transaction data
       - Blockchain: Public transaction data (bitcoin, ethereum) as partial analogue

    2. For each potential dataset found:
       - Document: Source, size, format, access method
       - Assess: Does it represent A2A commerce? Or is it human-only? Mixed?
       - Evaluate: Representativeness for fraud detection validation
       - Verdict: Viable as-is, viable with adaptation, or not viable

    3. If NO A2A datasets found:
       - Explicitly document: "No public A2A transaction datasets found"
       - List: What was searched and where
       - Explain: Why this gap exists (emerging domain, proprietary platforms)
       - Validate: This is a genuine gap, not a search failure

    4. For non-A2A datasets that might serve as partial analogues:
       - Document: Credit card fraud datasets, financial transaction datasets
       - Assess: What human patterns they contain (for comparison)
       - Note: Where they fall short for A2A validation

    Public datasets only per locked decision. No private API access assumptions.
  </action>
  <verify>
    1. Search completeness: Multiple sources checked with methodology documented
    2. Gap honesty: If no A2A datasets found, state explicitly with rationale
    3. Partial analogues: Non-A2A datasets documented with limitations noted
    4. Access verification: Don't claim dataset access without verifying link/access method
    5. Source documentation: All search results include specific sources and dates
  </verify>
  <done>Systematic search for public A2A transaction datasets complete. Results documented with sources. If gap exists, explicitly and honestly documented with rationale.</done>

<task type="auto">
  <name>Task 2: Specify synthetic data generation requirements</name>
  <files>analysis/synthetic-data-spec.md</files>
  <action>
    Specify synthetic A2A transaction data generation based on platform analysis and literature findings.

    1. Data requirements from platform analysis (Plans 01-01, 01-02):
       - Transaction patterns observed in OpenClaw/Moltbook documentation
       - Velocity capabilities (rate limits, transaction speed)
       - Social/reputation patterns (if applicable)
       - Attack chain behavioral patterns identified

    2. Data requirements from literature (Plan 01-03):
       - Established transaction patterns from multi-agent economics
       - Fraud patterns from nearest analogues
       - Human invariant benchmarks for comparison

    3. Synthetic data specification:
       - Schema: Transaction format (timestamp, sender, receiver, amount, type, metadata)
       - Scale: Number of transactions, time span, agent count
       - Patterns: Legitimate agent transactions, fraudulent patterns (by attack vector)
       - Labels: Fraud/no-fraud labels for validation
       - Variations: Different fraud types per taxonomy from literature

    4. Generation approach:
       - Use Python (pandas, numpy) for generation
       - Base patterns on platform-identified capabilities
       - Inject fraud patterns based on attack chains from Plans 01-01, 01-02
       - Include human transaction baseline for comparison

    5. Validation criteria:
       - Statistical realism: Distributions match expected patterns
       - Attack coverage: All attack vectors from taxonomy represented
       - Label accuracy: Clear ground truth for detection testing

    6. CRITICAL: Limitations section:
       - What synthetic data CAN'T capture (emergent behavior, real-world complexity)
       - Validation uncertainty: Results may not generalize
       - Real data prerequisite: What would be needed for full validation

    Every synthetic requirement must trace to platform analysis or literature finding.

    Note: This is specification only. Implementation occurs in later phases.
  </action>
  <verify>
    1. Platform grounding: Each synthetic requirement references platform analysis or literature
    2. Attack coverage: All attack vectors from taxonomy represented in specification
    3. Baseline inclusion: Human transaction patterns included for comparison
    4. Label clarity: Clear fraud/no-fraud ground truth specification
    5. Limitations comprehensive: What synthetic can't capture is explicitly documented
    6. Not implementation: Specification only; actual generation deferred to appropriate phase
  </verify>
  <done>Synthetic data specification complete with platform-grounded requirements, attack coverage, human baseline, and comprehensive limitations documentation. Specification-only; implementation deferred.</done>

    3. Synthetic validation strategy (if applicable):
